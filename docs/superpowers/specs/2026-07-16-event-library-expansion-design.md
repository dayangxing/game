# 事件库扩容与 LLM 叙事层设计规格

日期：2026-07-16
状态：待用户评审
目标分支：`dev`

## 1. 背景与目标

当前事件系统已经具备事件目录、触发条件、选项效果、冷却与结果解析能力，但事件数量与章节绑定还不足以支撑长期游玩。本次工作扩容为一套章节驱动的事件库，覆盖序章、炼气、筑基、金丹、雾隐秘境、飞升骗局、终局分支七个章节，共约 50 个正式事件。

本规格解决四个问题：

1. 玩家在每个章节都有明确的主线事件节奏，并能看到选择如何积累为终局差异。
2. 事件内容覆盖境界成长、NPC、宗门、秘境、寿元危机、突破失败和隐藏结局。
3. 事件规则保持服务器权威、可测试、可复现，LLM 保留场景化叙事、对白和局部变体空间。
4. 主线事件一次性推进，支线事件可重复但受冷却和递减收益控制，避免刷事件破坏节奏。

数值平衡、存档槽、成就与传记导出增强属于后续独立工作。本次只提供事件内容和运行时扩展所需的稳定接口，不重新设计这些系统。

## 2. 设计原则

### 2.1 规则与叙事分层

事件目录是唯一的规则来源。后端决定事件是否可见、选项风险、突破与资源变化、旗标、关系、宗门立场、后续事件以及事件是否完成。LLM 只负责将已确定的结果组织成符合当前角色和章节语境的文本。

LLM 可以读取：

- 当前章节标题、章节目标的玩家可见文本和当前场景；
- 玩家当前境界、公开状态、近期事件结果摘要；
- 事件的场景、情绪、NPC 角色和感官标签；
- 玩家选择对应的叙事意图；
- 后端已解析出的公开结果。

LLM 不可以读取或改变：

- 内部目标 ID、未公开的触发条件、结局 ID；
- 实际数值公式、概率和完整效果定义；
- 内部旗标名称及其写入方式；
- 服务器已经判定的成功、失败、消耗与奖励。

这样既保留“每次游玩像在和一个会记住前因后果的叙事者对话”的特色，也避免模型直接改数值或提前泄露谜底。

### 2.2 章节驱动而不是硬线性脚本

每个事件标注一个或多个章节 ID，并由当前章节、公开章节进度和事件触发条件共同筛选。主线事件可以有不同入口和选择，但最终汇入章节目标、NPC 关系或宗门立场等稳定状态，不维护一棵难以测试的无限分叉剧情树。

### 2.3 选择必须形成可感知差异

每个正式事件提供两个选择：一个低风险、收益稳定的选择，一个高风险、收益或线索更强但可能付出代价的选择。两个选择都必须有明确的后端效果或状态差异，不能只改变措辞。

## 3. 事件数据模型

在现有 `eventCatalog.js` 的数据驱动结构上扩展字段，不引入第二套事件格式。正式事件采用以下语义：

```js
{
  id: 'mist_bronze_bell',
  chapterIds: ['mist'],
  category: 'mystery',
  cadence: 'mainline',
  oneShot: true,
  cooldownTurns: 0,
  trigger: {
    realmAtLeast: '金丹初期',
    requiresFlags: ['mist_entry_unlocked']
  },
  narrativeContext: {
    scene: '雾隐秘境边缘',
    mood: '诡谲、克制',
    npcRoles: ['lin_shijie'],
    sensoryTags: ['铜铃', '白雾', '残碑']
  },
  choices: [
    {
      id: 'approach',
      label: '靠近铜铃',
      risk: 'medium',
      narrativeIntent: '主动追查铜铃来源',
      success: { text: '...', effects: [] }
    },
    {
      id: 'observe',
      label: '远处观察',
      risk: 'low',
      narrativeIntent: '保守记录秘境异象',
      success: { text: '...', effects: [] }
    }
  ]
}
```

字段约定如下：

- `chapterIds`：事件所属章节；未填写时仅允许兼容旧事件，新增正式事件必须填写。
- `category`：内容维度，如 `cultivation`、`npc`、`sect`、`mystery`、`lifespan`、`breakthrough`、`hiddenEnding`。
- `cadence`：`mainline` 或 `side`。主线事件参与章节进度；支线事件不直接完成章节目标。
- `oneShot`：主线默认为 `true`。支线默认为 `false`，但仍受冷却和重复收益规则约束。
- `cooldownTurns`：支线再次出现前需要经过的回合数；主线固定为 0，因 `oneShot` 防止重复。
- `narrativeContext`：仅供叙事层使用的场景素材，不承载规则。
- 每个选择必须继续使用现有效果解析结构，并增加 `narrativeIntent` 作为叙事提示。

主线的“汇流”通过现有或扩展后的效果类型完成，例如公开旗标、NPC 关系、宗门立场、业力和后续事件标记。事件不直接写入不可审计的自由文本状态。

## 4. 触发与重复规则

### 4.1 触发条件

在现有 `viewIds`、旗标、未来事件、宗门关系和业力条件上，增加以下可组合条件：

- `chapterIds`：当前章节必须属于事件的章节范围；
- `realmAtLeast`：境界达到指定门槛；
- `npcAffinityMin`：指定 NPC 关系达到门槛；
- `requiresSectPath`：已经选择指定宗门路线；
- `lifespanRatioMax`：当前寿元占初始寿元比例低于或等于阈值；
- `requiresBreakthroughFailure`：指定境界突破失败次数达到要求；
- `requiresEventResolved`：指定事件已完成；
- `forbidEventResolved`：指定事件尚未完成；
- `oneShot` 的完成判定：事件已完成时不再进入候选池。

条件全部采用 AND 语义；同一条件内部的数组仍按现有约定使用“任一满足”或“全部满足”，并在测试中明确。触发器只负责资格判断，不执行效果、不决定概率。

### 4.2 主线事件

主线事件的完成状态沿用服务器已有的事件结果记录或等价的已解决事件集合。事件完成后：

1. 该事件不再进入候选池；
2. 其选择效果保留在游戏状态中；
3. 章节目标由权威状态重新计算，而不是由客户端推进；
4. 需要下一事件链时，通过 `futureEvent` 或事件完成条件开放后续事件。

### 4.3 支线事件

支线事件可重复出现，但必须同时满足：未处于冷却、未在近期重复出现、当前资源能够支付效果。重复收益使用确定性衰减：首次完整奖励，第二次为标准奖励的 50%，第三次及以后为标准奖励的 25%，并设置下限为 1 点或 1 件可发放单位；线索旗标和关系变化不因衰减而丢失，但同一旗标不能重复写入。

递减次数和冷却状态必须属于服务器状态，不能由前端或 LLM 维护。若当前状态结构尚无对应字段，实施阶段添加最小的 `eventHistory` 元数据，而不把完整叙事文本塞入存档状态。

## 5. 事件库内容分配

目标规模为 50 个正式事件。数量是内容覆盖的验收基线，不限制未来继续扩容。

| 章节 | 数量 | 内容重点 | 章节出口 |
| --- | ---: | --- | --- |
| 序章 `prologue` | 6 | 命簿、寿元异常、初识 NPC、宗门入口、第一道雾隐线索 | 任一 NPC 初始关系与首个真相旗标 |
| 炼气 `qi` | 7 | 日常修炼、炼丹、寿元压力、突破准备、第一次突破失败、师友态度 | 炼气圆满与寿元痕迹确认 |
| 筑基 `foundation` | 7 | 筑基风险、失败后恢复、宗门试炼、长老分歧、关键 NPC 关系 | 筑基成功、宗门路线、关系线 |
| 金丹 `golden_core` | 7 | 结丹、宗门冲突、秘境资格、 NPC 选择、资源取舍 | 金丹成功、秘境入口、长老分歧 |
| 雾隐秘境 `mist` | 8 | 铜铃、残档、迷雾代价、青铜铃、天门秘钥、真相拼合 | 至少三条真相与关键秘钥线索 |
| 飞升骗局 `ascension_scam` | 8 | 天门契、账帖、飞升者记录、伪飞升者、契约代价、立场选择 | 契约确认、四条以上真相、契约立场 |
| 终局分支 `finale` | 7 | 最终抉择、代价兑现、NPC/宗门回收、失败重开提示、隐藏结局前置 | 正式结局或隐藏结局 |

每章的事件构成为 4–5 个推进章节目标的主线事件，加上 2–3 个可重复支线事件。主线事件之间允许多入口，但每章至少存在一条不依赖隐藏条件的可完成路径，避免玩家因错过单个事件而软锁。

内容维度覆盖要求：

- 境界：每章至少有一个与当前境界或突破准备相关的事件；
- NPC：四条以上人物关系线，各自至少拥有一次主动选择和一次关系回收；
- 宗门：至少三次宗门态度变化，并在筑基、金丹、终局各有一次可感知回收；
- 秘境：雾隐章节至少六个直接相关事件，且线索可以通过不同选择汇流；
- 寿元危机：至少四个事件使用寿元阈值或寿元代价，其中一个属于终局前预警；
- 突破失败：炼气、筑基或金丹至少各有一条失败后可恢复的事件路径；
- 隐藏结局：至少两条独立前置组合，不在普通事件文案中直接泄露结局名称。

## 6. 分支汇流与隐藏结局

选择不以“唯一正确答案”设计，而是改变后续可见内容、关系和代价。高风险选择可以更快获得真相、宝物或功法，但可能损失寿元、宗门关系或增加业力；低风险选择获得较少即时收益，但保留更多后续选择空间。

分支汇流使用三类可追踪状态：

1. 真相旗标：记录玩家已经确认的关键事实；
2. 关系与立场：记录 NPC 关系、宗门路线和契约立场；
3. 代价状态：记录寿元、业力、突破失败次数和必要的负面债务。

终局只读取这些稳定状态和章节目标，不读取事件文案。隐藏结局至少需要“真相组合 + 立场或关系组合 + 代价条件”中的两类，且在未满足前置时不得从 API 或 LLM 上下文泄露结局 ID。

## 7. 运行时与 LLM 数据流

一次事件回合的权威流程如下：

```text
当前游戏状态
    -> 触发器筛选章节事件
    -> 冷却/一次性/近期重复筛选
    -> 后端发布两个公共选项
    -> 玩家选择
    -> 后端校验并解析效果
    -> 写入状态、事件历史和章节进度
    -> 生成公开结果摘要
    -> LLM 根据叙事上下文生成场景与对白
```

LLM 的请求上下文只携带经过裁剪的公开快照和 `narrativeContext`。后端结果中的数值变化以结构化摘要提供给前端；叙事文本不能反向触发效果。若 LLM 调用失败，系统仍使用事件目录中的确定性 `success.text` 或失败文本完成回合，确保事件系统可离线运行。

为了保留特色，叙事层应重点利用：

- 当前 NPC 关系与玩家此前的选择摘要；
- 章节情绪和感官标签；
- 选择的叙事意图与真实结果之间的张力；
- 已公开真相的组合，而不是机械重复事件标题。

这些输入只影响表达、视角、对白和局部细节，不改变可验证的游戏状态。

## 8. 测试与验收标准

### 8.1 单元测试

- 章节、境界、NPC、宗门、寿元和突破失败触发器分别覆盖满足与不满足场景；
- 主线事件完成后不会再次进入候选池；
- 支线事件在冷却期内不可见，冷却后可见；
- 支线重复收益按 100% / 50% / 25% 递减，并受下限保护；
- 两个选择都能产生可观测的状态差异；
- 资源不足时不会产生部分效果或负数资源；
- 事件完成后章节目标由状态正确推进；
- 隐藏结局前置不满足时，候选事件和叙事上下文不暴露结局 ID。

### 8.2 内容完整性测试

- 正式事件数量为 50；
- 七章数量分别为 6、7、7、7、8、8、7；
- 每个正式事件都有 `chapterIds`、`category`、`cadence`、`narrativeContext` 和两个选择；
- 每个章节至少有一条可完成的主线路径；
- 事件 ID、旗标、后续事件引用和 NPC/宗门引用无悬空项；
- 所有主线事件均有一次性语义，所有支线事件均有冷却或近期重复保护。

### 8.3 集成与回归测试

- 服务器 API 仍只暴露公共事件选项，不暴露内部效果定义和隐藏触发器；
- 事件结果、流式故事和章节进度在同一回合保持一致；
- 旧事件和旧存档可以通过默认字段兼容运行；
- 前端显示两个选择、风险提示和结果摘要，不承担事件判定；
- 完整测试集保持通过，新增测试覆盖事件目录、触发器、重复规则、LLM 上下文边界和 API 脱敏。

### 8.4 人工游玩验收

至少验证三条路线：

1. 稳健路线：多数选择低风险，正常完成主线并进入普通结局；
2. 冒险路线：多次选择高风险，经历至少一次突破失败但仍可继续；
3. 真相路线：完成关键线索组合，触发隐藏结局前置且不提前显示结局名称。

每条路线都要观察到 NPC、宗门、寿元或线索状态的差异，而不是只得到不同的叙事句子。

## 9. 实施边界与并行策略

本功能会频繁修改 `eventCatalog.js`、`triggerMatcher.js`、事件选择器和 API 脱敏边界，核心规则不适合多人或多代理同时改写。实施顺序应为：先串行稳定事件模型和触发器，再按章节拆分内容数据；内容录入只有在共享 schema 和校验测试完成后才适合并行。

推荐的提交粒度为：

1. 触发器与事件元数据兼容层；
2. 重复、冷却和递减收益规则；
3. 目录完整性校验与测试夹具；
4. 七章事件内容，按章节串行或互不重叠文件分批提交；
5. LLM 叙事上下文与 API 脱敏；
6. 全量回归与人工路线验收。

所有提交只进入 `dev`。除非用户明确要求，不提交或推送 `main`。
## 10. 非目标

本次不实现以下内容：

- 新的数值平衡表和完整经济系统重做；
- 多存档槽、云端账号同步或存档格式迁移；
- 成就系统的完整 UI 与跨周目统计；
- 让 LLM 自主生成、注册或执行新的事件规则；
- 将事件改造成不可审计的自由图节点系统。
