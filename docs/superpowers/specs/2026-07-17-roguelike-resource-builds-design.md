# Roguelike 功法与宝物构筑设计

**日期：** 2026-07-17
**状态：** 已确认设计，待实现

## 目标

把现有固定的功法、宝物奖励扩展为事件驱动的 Roguelike 构筑系统：玩家在一局中通过有剧情来源的奇遇获得资源，从三张候选卡中选择一张，资源标签形成流派共鸣；死亡或终局后清空本局构筑，但保留发现记录和元进度。

## 范围

本轮只处理功法、宝物和它们与奇遇事件的连接，不新增宗门系统。

包含：

- 功法目录从现有 2 门扩展到 8 门。
- 宝物目录从现有 2 件扩展到 8 件。
- 资源品阶、类型、适用境界、流派标签和数值加成。
- 事件绑定资源池，相关奇遇结束后提供三选一。
- 同一局不重复提供已经拥有的功法或宝物。
- 流派共鸣和真实数值计算。
- 本局资源与死亡后保留的发现/解锁状态分离。
- 前端资源卡、资源候选卡和共鸣展示。

不包含：

- 宗门目录、宗门切换和宗门关系重构。
- LLM 动态生成资源或数值。
- 突破成功后的自动掉落。
- 新的战斗系统；`damageReduction` 本轮只用于降低突破失败的气血损失。
- 元进度商店或复杂的局外天赋树。

## 设计原则

1. 规则系统决定资源、候选池、数值和获得条件，LLM 只描述已经发生的资源发现。
2. 每个资源必须有可追溯的事件来源；普通修炼、突破、寿元消耗和章节推进不能凭空产生资源。
3. 随机结果由本局种子、事件 ID 和回合共同决定，同一存档重试得到相同候选。
4. 资源选择是构筑决策，不是单纯提高数值；不同标签要支持明确的流派方向。
5. 旧资源 ID 和旧存档字段保持兼容，不能因为扩容要求玩家重开已有存档。

## 资源目录

### 统一字段

功法和宝物目录条目使用相同的基础结构：

```js
{
  id: 'stable_resource_id',
  name: '玩家可见名称',
  grade: '凡品|良品|上品|传说',
  type: '心法|身法|剑诀|炼体|神识|养元|香|护具|法器|丹炉|命器|灵材',
  realmAtLeast: '炼气一层',
  tags: ['养元', '木灵'],
  description: '简短的玩家可见描述。',
  detail: '更具体的效果和来历描述。',
  bonuses: {
    cultivationGain: 0,
    breakthroughChance: 0,
    maxHealth: 0,
    maxLifespan: 0,
    damageReduction: 0
  }
}
```

未使用的加成字段不写入目录；加成值由规则系统求和，不能由前端或模型解释。

## 资源池与事件来源

资源池由规则目录定义，事件只引用池 ID 和玩家可见的发现理由：

```js
{
  id: 'mistRelics',
  label: '雾隐遗物',
  tags: ['雾隐', '神识'],
  resourceIds: [
    'mist_step',
    'calm_lotus_incense',
    'bronze_bell_fragment',
    'mist_veil'
  ],
  narrativeReason: '你从雾灯、残碑与青铜铃的回声中辨认出三件可以带走的遗物。'
}
```

首批资源池：

- `mistRelics`：雾隐秘境、铜铃、雾灯和残碑相关奇遇。
- `scriptureArchive`：藏经阁、残卷、长老指点和石壁悟法相关奇遇。
- `alchemyFinds`：药圃、丹房、药炉和命火调养相关奇遇。
- `beastSpoils`：妖兽、巡山、遗迹护卫和炼体相关奇遇。
- `ancientRuins`：古修遗府、星盘、命灯和飞升旧档相关奇遇。

每个池至少提供四个候选资源，便于过滤本局已拥有的条目后仍生成三张卡。资源池不足时，先从同标签的相邻池补足；仍不足时提供明确标注的灵石补偿卡，不伪装成新的功法或宝物。

### 事件奖励流程

资源奇遇的选择效果使用新的 `resourceDraft` 规则效果，而不是直接把资源塞进背包：

```js
{
  type: 'resourceDraft',
  poolId: 'mistRelics',
  reason: '你在雾隐石档库中找到了可以带走的遗物。'
}
```

事件结算时：

1. 先照常结算事件的属性、旗标、关系和时间结果。
2. 记录事件叙事和资源发现理由。
3. 由服务端用 `seed + eventId + turn` 生成三张候选卡。
4. 将候选卡作为待选择的 `resourceDraft` 保存到游戏状态。
5. 前端展示三张卡，玩家选择一张；这次选择不消耗额外回合和寿元。
6. 选中的资源加入 `game.techniques` 或 `game.treasures`，同时写入本局获取记录。

普通事件不带 `resourceDraft` 就不会生成候选。突破相关逻辑不创建 `resourceDraft`，因此突破成功或失败都不会莫名出现资源。
现有固定功法/宝物效果只保留给明确的剧情锚点事件；这些事件同样必须写入 `acquisitionLog`，并在叙事中说明资源来源。

## 本局状态与元进度

### 本局状态

继续使用现有字段作为当前有效构筑：

```js
game.techniques
game.treasures
```

新增：

```js
resourceRun: {
  pendingDraft: null,
  activeResonances: [],
  resolvedDraftIds: [],
  acquisitionLog: []
}
```

`acquisitionLog` 记录 `{ kind, resourceId, eventId, eventTitle, turn }`，用于传记、前端来源提示和失败总结；目录对象本身不写入运行时来源，避免污染共享目录。

### 元进度

```js
metaProgress: {
  discoveredTechniques: [],
  discoveredTreasures: [],
  unlockedTechniques: [],
  unlockedTreasures: [],
  runCount: 0,
  bestChapter: null
}
```

本轮所有资源都可以由对应奇遇池发现；首次选择后加入发现和解锁记录，后续新局继续保留。`unlocked*` 与 `discovered*` 分开保存，便于未来增加解锁条件，但本轮不增加元进度商店。

寿元终章、正式结局和手动重开在清空本局资源前，先把本局获取过的资源合并进元进度，并更新 `runCount`、`bestChapter`。旧存档缺少这些字段时按空数组和零值迁移。

## 数值与共鸣

### 加成落点

- `breakthroughChance`：直接参与现有突破成功率计算。
- `maxHealth`、`maxLifespan`：沿用现有奖励同步逻辑，重新计算上限并夹紧当前值。
- `cultivationGain`：只加到正向 `player.cultivationProgress` 规则效果，不影响失败回退和负向消耗。
- `damageReduction`：在突破失败时按 `floor(damageReduction / 2)` 减少气血损失，最低保留 1 点损失。

### 共鸣规则

共鸣按当前功法和宝物的全部标签计数，每种标签只激活最高层级：

| 标签 | 2 件共鸣 | 3 件共鸣 |
| --- | --- | --- |
| 雷法 | `breakthroughChance +2` | `breakthroughChance +4` |
| 养元 | `maxLifespan +4` | `maxLifespan +8` |
| 炼体 | `maxHealth +6` | `maxHealth +12` |
| 雾隐 | `damageReduction +2` | `damageReduction +5` |
| 神识 | `breakthroughChance +2` | `breakthroughChance +4` |

共鸣加成进入 `calculateDerivedBonuses`，与资源加成使用同一套同步和测试路径。前端显示当前激活的标签、层级、效果和还差几件触发下一层。

## API 与前端边界

服务端保留资源目录、候选池、随机种子和待选状态的权威性。公开状态只输出玩家需要选择和展示的字段，不输出池内规则、随机种子或内部事件字段。

资源候选动作使用现有每日行动/回合结算通道的公开动作格式，额外标记 `category: 'resource'`；客户端只能提交候选动作 ID，不能提交资源 ID、加成或来源。

前端增加：

- 资源候选区域：显示来源事件、发现理由、三张候选卡和选择按钮。
- 功法/宝物卡的品阶、类型、标签、加成和获取来源。
- 当前构筑的共鸣面板。
- 终局/失败总结中的本局资源和新增发现。

## 错误与兼容

- 非法资源 ID、非法资源池 ID、已拥有资源重复选择都由服务端拒绝。
- 待选择草稿不存在或已消费时返回可理解的刷新错误，不改变游戏状态。
- 旧存档中的字符串资源 ID继续通过现有目录归一化；未知旧条目保留原数据并不参与新的共鸣计算。
- 模型输出不能创建、替换或升级资源；叙事上下文只读取当前已获得资源。

## 测试验收标准

- 资源目录存在 8 门功法、8 件宝物，ID 唯一且所有加成字段都在白名单内。
- 所有资源池至少有 4 个资源，事件引用不存在悬空池 ID。
- 给定相同种子、事件 ID 和回合时，候选顺序完全一致。
- 已拥有资源不会再次出现在同一局候选中；资源池不足时走明确补偿路径。
- 只有携带 `resourceDraft` 的奇遇能创建候选；突破和普通修炼测试确认不会创建候选。
- 选择候选后只加入对应资源集合、来源日志和元进度，不额外消耗回合或寿元。
- 共鸣阈值、资源加成、修为收益和突破失败减伤都有单元测试。
- 寿元终章、正式结局和重开会清空本局资源并保留发现/解锁/局数进度。
- 前端测试覆盖候选卡、资源来源、加成文本和共鸣面板，且不暴露内部 ID、池规则或随机种子。
- 完整 `npm test` 通过，`git diff --check` 通过。
