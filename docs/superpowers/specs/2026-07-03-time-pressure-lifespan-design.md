# 时间压力与寿元大限系统设计

## Goal

把《问道浮生》的连续剧情从“像一天内不断发生的流水剧情”升级为“带时间跨度、寿元压力和大限目标的单机修仙循环”：

- 每次主页推进都会消耗一段明确时间，而不是默认瞬时发生。
- 境界越高，闭关、探秘、推演剧情消耗的现实时间越长。
- 寿元成为真正的失败压力：玩家必须在寿元大限之前推进主线、突破境界、揭开伏笔。
- 突破成功能增加寿元和寿元上限，突破失败、负面事件、秘境反噬会损失寿元。
- 大模型继续负责连续剧情和选项意图，具体时间、寿元、修为和状态数值由后端规则结算。
- 前端强化“时间过去了”和“命火在减少”的反馈，让玩家能感到选择有代价。

第一版目标是增加紧迫感和策略权衡，不是做硬核 Roguelike。玩家应该感觉“乱选会输，规划能活”，而不是“随便点也能过”。

## Current Context

项目当前已经具备这些基础：

- `src/engine.js` 已有 `calendar`，旧原型每回合推进 1 个月。
- `backend/src/domain/progression.js` 已有按境界计算寿元消耗的 `calculateLifespanCost` 和突破失败代价。
- `resolveBreakthrough` 已能区分成功和失败，失败会扣气血、寿元和修为进度。
- 连续剧情 director 已经采用“模型生成 scene/choices/effectHints，后端结算数值”的边界。
- 前端主页已经显示寿元、境界、历史行为和顶部时间。
- `storyMemory` / 天机录已经记录近期行动、伏笔和上下文。

当前短板：

- 连续剧情没有明确时间跨度，玩家感觉所有事件都发生在同一天。
- 寿元消耗虽然存在，但没有和“历时多久”绑定，感知不强。
- 突破成功没有明确“续命”奖励，寿元系统缺少正反馈。
- 负面事件对寿元的威胁不够突出。
- 模型上下文没有把“大限压力、剩余寿元比例、已耗时间”作为导演条件。

## Product Decision

采用“规则驱动时间压力，模型负责叙事”的方案。

后端规则层负责：

- 判断本回合行动类型。
- 根据境界和行动类型计算历时。
- 推进日历。
- 根据历时、境界和事件影响扣寿元。
- 处理突破成功的寿元增长。
- 处理寿元归零后的结局状态。
- 输出玩家可读的 `timeDelta`、`lifespanDelta` 和变化摘要。

大模型负责：

- 根据当前时间、寿元压力、近期剧情和伏笔写连续剧情。
- 在关键节点生成选择。
- 对选择给出模糊影响，如耗时偏短、耗时偏长、寿元风险高、修为提升、伏笔推进。
- 不输出具体数值，不直接修改寿元、境界、道具或功法。

前端负责：

- 显示本回合经过了多久。
- 显示寿元变化和剩余大限。
- 在寿元低时强化警示。
- 在历史行为中展示“历时 X、寿元 -Y、修为 +Z”等玩家可读信息。

## Non Goals

第一版不做以下内容：

- 不做离线挂机、时间自动流逝或真实时间倒计时。
- 不让大模型直接决定寿元数值。
- 不做复杂经济系统、宗门年度排期或完整日程表。
- 不做多结局树，只做寿元归零的“命簿终章”失败态。
- 不做难度选择界面，先用一套默认平衡参数。
- 不重写整个事件库，只扩展连续剧情和现有规则层。

## Time Model

时间以“月”为后端最小单位。

```js
time: {
  elapsedMonths: 37,
  lastDeltaMonths: 3
}
```

`calendar` 继续保留前端可读结构：

```js
calendar: {
  year: 6,
  season: '秋',
  month: 8
}
```

`elapsedMonths` 是权威计算字段，`calendar` 是展示字段。这样避免只靠 `year/month` 推导时出现跨年误差。

### Realm Time Scale

基础耗时按境界增长：

| 境界 | 普通推进 | 修炼/闭关 | 探秘/远行 | 突破 |
| --- | ---: | ---: | ---: | ---: |
| 炼气 | 1 月 | 3 月 | 2 月 | 6 月 |
| 筑基 | 3 月 | 6 月 | 4 月 | 12 月 |
| 金丹 | 6 月 | 12 月 | 8 月 | 24 月 |
| 元婴 | 12 月 | 24 月 | 18 月 | 48 月 |

设计理由：

- 低境界推进快，玩家能较快进入核心循环。
- 高境界闭关时间显著拉长，修为越高越有“大修士一闭关数年”的感觉。
- 高境界寿元上限更高，但每次行动花费也更大，避免后期完全无压力。

### Action Time Category

后端把每轮推进归类为：

- `story`：普通连续剧情，默认耗时较短。
- `cultivation`：闭关、修炼、稳固境界，耗时较长。
- `explore`：秘境、后山、远行、追查线索，中等耗时并可能有额外风险。
- `social`：拜见长老、询问 NPC、宗门事务，耗时较短。
- `craft`：炼丹、整理行囊、准备资源，中等耗时。
- `breakthrough`：突破，耗时最长，收益和风险都最高。

模型可以通过 effectHints 给出模糊时间影响：

```json
{ "target": "time", "direction": "up", "intensity": "small" }
```

后端把它解释成额外耗时，不接受模型输出具体月份。

## Lifespan Model

寿元继续使用年为单位展示，但内部可以接受小数或按月累计。第一版为了减少 UI 复杂度，建议仍用整数年展示，并在后端按月换算后四舍五入到至少 1 年的可见变化。

推荐状态字段：

```js
player: {
  lifespan: 92,
  maxLifespan: 154
},
timePressure: {
  elapsedMonths: 37,
  lastDeltaMonths: 3,
  lastTimeLabel: '三月',
  lastLifespanCost: 1,
  lifespanRatio: 0.59,
  warningLevel: 'steady'
}
```

### Base Lifespan Cost

每次行动的寿元成本由两部分构成：

```text
timeCostYears = ceil(deltaMonths / 12)
realmPressure = existingRealmLifespanCost
lifespanCost = max(1, timeCostYears + realmPressure - lifeSeedReduction - bonusReduction)
```

设计要点：

- 即使普通剧情只过 1 个月，也至少有基础寿元压力。
- 高境界 `realmPressure` 更高，体现“高境界时间过得更快、维持道体更耗命火”。
- `lifeSeed` 和宝物可以减免，但不能把正式行动降到 0。

### Negative Events

负面事件不只是扣气血，也可能额外扣寿元：

- 命火反噬：寿元额外下降。
- 秘境侵蚀：寿元下降，可能换取伏笔推进。
- 飞升契印：寿元异常流失，但推动主线真相。
- 心魔拖延：耗时增加，寿元间接下降。
- 重伤闭关：增加后续修复耗时。

模型只给模糊影响：

```json
{ "target": "lifespan", "direction": "down", "intensity": "medium" }
```

后端根据境界、命元、宝物和当前寿元比例换算为实际数值。

## Breakthrough And Longevity

突破必须成为“续命”的核心手段。

### Success Rewards

突破成功时增加寿元和上限：

| 突破类型 | 当前寿元 | 寿元上限 |
| --- | ---: | ---: |
| 炼气小层 | +1 到 +2 年 | +0 到 +1 年 |
| 炼气九层到筑基 | +20 年 | +30 年 |
| 筑基小境界 | +5 到 +8 年 | +4 到 +6 年 |
| 筑基到金丹 | +50 年 | +80 年 |
| 金丹小境界 | +12 到 +18 年 | +10 到 +15 年 |
| 金丹到元婴 | +120 年 | +180 年 |

第一版可以先实现到金丹，元婴参数预留。

### Failure Costs

突破失败继续使用现有失败代价，并增加时间损失：

- 扣气血。
- 扣寿元。
- 扣修为进度。
- 推进突破耗时。
- 写入历史行为：`突破耗时一年，寿元 -X，修为回退。`

失败不应该总是灾难，但连续失败会显著接近大限。

### Risk Feedback

突破面板需要明确告诉玩家：

- 目标境界。
- 成功率。
- 预计耗时。
- 失败代价。
- 成功可获得的寿元增长范围。

例如：

```text
尝试突破 -> 筑基初期
成功率 43%
预计耗时 1 年
成功：寿元上限约 +30，当前寿元约 +20
失败：气血受损、寿元折损、修为回退
```

## Story Director Changes

连续剧情 director 的上下文需要新增：

```json
{
  "timePressure": {
    "calendarLabel": "玄历七年 秋 第三月",
    "elapsedYears": 4,
    "remainingLifespan": 42,
    "maxLifespan": 116,
    "lifespanRatio": 0.36,
    "warningLevel": "strained",
    "lastDeltaTime": "三月",
    "lastLifespanCost": 1
  }
}
```

prompt 约束：

- 必须承认时间流逝，不要让连续剧情都像同一天。
- 当寿元低于 45% 时，叙事应出现命火、白发、心悸、闭关成本等压力。
- 当寿元低于 20% 时，选项应更强调取舍：冒险推进主线、闭关续命、求助 NPC。
- 不得输出具体扣减数值。
- 可以暗示某选择“耗时更久”“风险更高”“可能推动伏笔”。

## Backend Architecture

新增建议模块：

```text
backend/src/domain/time/
  calendar.js
  timeCost.js
  timePressure.js
```

职责：

- `calendar.js`：按月份推进日历，格式化时间。
- `timeCost.js`：根据境界、行动类型、effectHints 计算 `deltaMonths`。
- `timePressure.js`：应用寿元扣减、生成 warningLevel、判断寿元归零。

与现有模块的关系：

- `progression.js` 保留突破成功率和突破失败代价，但突破结算调用时间系统。
- `effectHints.js` 增加 `time` target，并把时间影响转成规则层参数。
- `app.js` 在 `resolveDirectorTurn` 和普通行动结算后统一调用时间推进。
- `storyMemory.js` 记录每回合的时间标签和寿元压力。

推荐统一输出：

```js
timeResult: {
  deltaMonths: 6,
  label: '半年',
  lifespanCost: 2,
  warningLevel: 'strained',
  game
}
```

## Frontend UX

主页强化三类信息。

### Top Status

顶部时间显示从简单日期升级为：

```text
玄历七年 秋 第三月  |  余寿 42年 / 大限 116年
```

或者在寿元卡片中显示：

```text
寿元 42/116
本回合历时 三月
命火渐弱，行事务必克制。
```

### History Cards

历史行为卡片展示：

```text
闭关修炼
历时三月 · 寿元 -1 · 修为 +8
```

负面事件展示：

```text
雾中契印
历时半年 · 寿元 -4 · 气血 -12 · 伏笔推进
```

突破展示：

```text
突破成功
历时一年 · 寿元 +20 · 寿元上限 +30 · 境界 筑基初期
```

### Warning States

按寿元比例显示不同状态：

- `steady`：> 60%，命火尚盛。
- `strained`：35% 到 60%，命火渐弱。
- `danger`：15% 到 35%，大限逼近。
- `critical`：<= 15%，命火将熄，建议 UI 用更强提示。
- `ended`：<= 0，进入命簿终章。

## Death / End State

寿元归零后进入：

```js
game.ending = {
  type: 'lifespan_exhausted',
  title: '命簿终章',
  body: '命火在最后一夜熄灭，未解伏笔仍悬于天门之后。'
}
```

前端行为：

- 禁用 `继续` 和普通选择。
- 显示“查看传记”和“重开”。
- 天机录保留本局摘要、未解伏笔和最后时间。

第一版不做复活和续命结局树，未来可以加特殊道具或 NPC 牺牲线。

## Balance Targets

第一版默认目标：

- 正常规划的玩家应能在 40 到 70 次主页推进内到筑基，并明显感到寿元压力。
- 完全乱选高风险选项，应该在筑基前或筑基初期有较高失败概率。
- 成功筑基后压力暂时缓解，但金丹前再次变紧。
- 低风险选择安全但推进慢，高风险选择能推进主线或获得资源但消耗寿元。
- 修为提升不能只靠点击继续，玩家需要在修炼、探秘、社交和突破之间取舍。

## Testing Strategy

需要覆盖：

- 日历推进跨月、跨年、季节变化。
- 不同境界和行动类型的耗时。
- 寿元扣减不低于 1，且命元和宝物能减免。
- 突破成功增加寿元和寿元上限。
- 突破失败推进时间、扣寿元、扣气血、回退修为。
- 寿元归零进入 ending，后续普通行动被拒绝。
- director prompt 包含 timePressure，并禁止模型输出具体数值。
- 前端历史行为显示历时和寿元变化。
- 前端不显示 `deltaMonths`、`timeCost`、`effectHints` 等内部字段。

## Rollout Plan

第一阶段：规则层

- 新增时间模块。
- 接入普通行动和 director 回合。
- 接入突破成功/失败。
- 加 ending 状态。

第二阶段：模型上下文

- prompt 加 timePressure。
- effectHints 增加 `time` target。
- storyMemory 记录时间和大限压力。

第三阶段：前端显示

- 顶部和寿元卡片强化时间压力。
- 历史行为显示历时与寿元变化。
- 寿元归零显示命簿终章。

第四阶段：平衡调整

- 调整境界耗时表。
- 调整突破增寿表。
- 调整负面事件寿元风险。

## Open Decisions

本 spec 第一版暂定：

- 时间内部按月计算，UI 主要展示月/年。
- 第一版只做寿元归零失败态，不做复杂结局树。
- 第一版默认难度偏中等，不加难度选择。

后续可以再讨论：

- 是否加入“续命丹”“夺舍”“NPC 牺牲”这类特殊续命线。
- 是否给主线设置硬年份节点，比如雾隐秘境每 12 年开启一次。
- 是否在二周目加入更高压力的难度模式。
