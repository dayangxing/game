# PC 与移动端响应式布局设计

## 目标

让问道浮生前端适配常用 PC、平板和移动端尺寸，同时保留现有桌面布局、导航和按钮行为。移动端不再依赖横向滚动访问页面，也不把次要操作按钮全部挤在顶部。

## 范围

- 启用 `desktop`、`tablet`、`mobile` 三种布局模式。
- 使用 `1200px` 和 `768px` 作为主要布局断点。
- 使用 `900px` 作为顶部次要操作折叠断点。
- PC 保持顶部显示“指引 / 存档 / 传记 / 重开 / 随机”。
- 平板和手机将这些次要操作放进右上角“菜单”按钮展开的面板。
- 保留现有按钮 ID，避免改变存档、传记、重开、随机和指引的业务监听。
- 移动端主内容改为单列，卡片、表格和操作区域允许收缩到视口宽度。
- 不在本次任务中改动后端、游戏数值、API、数据存储或主线内容。

## 响应式行为

### 桌面端（宽度 >= 1200px）

- 使用当前宽屏顶部栏和内容卡片布局。
- 顶部次要操作按钮直接可见。
- 页面最大宽度保持 1500px，不设置会制造横向溢出的最小页面宽度。

### 平板端（宽度 768px - 1199px）

- 顶部栏允许换行，品牌、主导航和操作区按可用宽度重排。
- 次要操作在宽度 <= 900px 时进入菜单；900px 以上仍可直接显示。
- 双列内容卡片在空间不足时降为单列，主内容不超过视口宽度。

### 移动端（宽度 < 768px）

- 页面使用单列流式布局，取消 `body` 和 `.app` 的固定最小宽度。
- 主导航保留四个入口，采用可触摸的紧凑排列；底部固定导航不在本次范围内，避免重复维护导航状态。
- 顶部次要操作只显示“菜单”按钮，点击后展开右上角面板。
- 菜单项目每项至少 44px 高；“重开”使用危险色并与普通操作分组。
- 点击菜单项目后收起菜单；点击面板外部或按 `Escape` 也收起菜单。
- 面板按钮继续使用原有 `guideBtn`、`saveBtn`、`exportBtn`、`resetBtn`、`sampleBtn` ID。

## 菜单交互

顶部操作区增加一个 `utilityMenuBtn` 和 `utilityMenuPanel`。菜单按钮通过 `aria-expanded` 表达展开状态，通过 `aria-controls` 指向面板；面板通过 `aria-hidden` 表达可见状态。桌面端面板使用 `display: contents`，使原有按钮保持当前布局；折叠断点以下面板变成绝对定位的纵向菜单。

菜单开关只负责视图状态，不改变任何游戏业务逻辑。现有各业务按钮监听继续绑定在原按钮上，菜单关闭逻辑通过统一的 `closeUtilityMenu()` 辅助函数复用。

## 测试与验收

- 单元测试覆盖三种宽度对应的布局模式和最小宽度契约。
- 前端结构测试覆盖菜单按钮、面板、ARIA 属性和原业务按钮 ID 保留。
- 运行完整 `node --test`。
- 浏览器验证 `390x844`、`768x1024`、`1280x800`、`1440x900` 四种尺寸。
- 移动端和平板端的 `document.documentElement.scrollWidth` 不得大于视口宽度加 1px。
- 桌面端顶部按钮可见，折叠端菜单按钮可见，菜单展开后五个原操作按钮可见且能正常触发。
- 不修改 `main` 分支，不推送远程分支。

## 取舍

本次选择“顶部菜单面板”而不是横向滚动或独立移动端 HTML。这样可以减少重复结构，保留现有业务监听，同时让移动端顶部保持简洁。主导航暂时不改为底部导航，以控制范围并避免与现有 hash 路由和页签状态产生重复交互。
